# 설계 결정 문서

본 문서는 개발 과정에서 결정한 내용을 정리한 문서입니다.

## DB — mysql or postgresql 선택

postgresql의 JSONB 타입 컬럼이 프로젝션 기반 설계에 잘 맞다고 판단했습니다.

json 내부의 각 필드에 인덱스 설정이 가능하기 때문에, 새로운 타입의 이벤트가 추가되더라도 DB 정규화에 대한 고민을 줄이고 비즈니스 로직 구현에 집중 가능합니다.

json 내부의 각 필드는 외래키 설정이 불가능하다는 제약이 있지만, 외래키 미사용을 결정하였기 때문에 감수할 수 있는 트레이드오프라고 판단하였습니다.
아래 문단에서 관련 내용을 다룹니다.

## 외래키 미사용

- 이벤트 기반 설계에서 insert가 빈번하므로, 외래키 적용시 레코드 검증과정이 추가되어 오버헤드가 발생합니다.
- projection 방식 설계라는 점에서 event 테이블이 single source of truth 역할을 하고 있습니다. 따라서 foreign key 가 보호해야 할 진실이 없습니다.

트레이드오프
- 잃은 것: 메세지 테이블의 경우, 잘못된 session_id 의 입력으로 인한 데이터 정합성 문제 발생 가능성 (애플리케이션 단에서의 버그)
- 완화 방법: 단일 진실인 이벤트 테이블을 replay 하여 메세지의 완전한 복구 가능
- 얻은 것: JSONB 타입을 이용한 유연한 도메인 확장 및 insert 성능

## docker-compose 설정

postgresql 실행에 docker compose 를 이용합니다.
named volume 을 이용해서 영속화된 정보가 도커 볼륨에 저장되도록 구현했습니다.
호스트 os 와 디렉토리 경로를 연동하는 bind mount 방식과 구분되므로, 테스트 실행자의 로컬 기기 환경과 무관하도록 하여 편의성을 제공합니다.

## 인증 로직은 x-user-id 헤더로 대체

인증이 완료된 사용자의 id가 x-user-id 헤더에 주입된 상태라고 가정합니다.

## 앱이 충족해야 할 비즈니스적 상황 가정

스프링부트 인스턴스 하나만으로 서비스를 시작하는 초기 비즈니스 앱이라고 가정했습니다.
과제 4.1에 명시된 필수 구현을 충족시킬 수 있는 가정입니다.

채팅 UX 결정에는 카카오톡의 작동 방식을 부분적으로 참고했습니다.

## 기능 요구사항 정의 및 이해

### 세션의 상태와 이벤트 종류에 대한 이해

과제에서 세션의 상태 3가지를 제시하고 있습니다. (진행중/중단/완료)
이벤트/메세지의 종류 5가지를 제시하고 있습니다. (message, join, leave, disconnect, reconnect)

이를 다음과 같이 해석했습니다.

세션의 상태
- 진행중: 두 사람 모두 웹소켓 연결이 활성화된 상태
- 중단: 둘 중 한 사람 이상의 웹소켓 연결이 끊어진 상태
- 완료: 둘 중 한 사람 이상이 명시적으로 퇴장한 상태

이벤트/메세지 종류
- message: 사용자가 전송한 채팅 메세지
- join: 사용자의 명시적인 세션 입장
- leave: 사용자의 명시적인 세션 퇴장
- disconnect: 사용자의 웹소켓 연결이 끊어짐 (명시적인 퇴장과 구분)
- reconnect: 연결이 끊어진 사용자의 웹소켓이 재연결됨 (명시적인 입장과 구분)

### 세션의 중단 상태에 대한 이해

세션에는 '중단'이라는 상태값이 존재합니다.
중단 상태에서 어떠한 기능구현을 해야할지 이해하는데 다소 어려움이 있었습니다. 일반적으로 카카오톡과 같은 채팅에는 중단이라는 상태가 명시적으로 나타나지 않기 때문입니다.
이를 다음과 같이 이해했습니다.
- 참가자의 연결이 일시적으로 끊어진 상태를 보여주는 view 기능이다.
- 채팅 기능에는 영향을 주지 않는다. (채팅 상대방은 정상 연결 상태이므로 메세지 전송 가능)

## 테이블 설계

### 메세지 테이블과 이벤트 테이블 이원화

event projection 방식을 깊게 적용한다면, 메세지 테이블 없이 이벤트 테이블 만으로도 기능 구현이 가능합니다.

다만 이 경우에는 채팅이 길어질 경우에 모든 히스토리를 replay 하는 비용이 발생하므로, 운영 환경에서 사용자 UX 향상을 위해 메세지 테이블과 이원화하였습니다.

### 이벤트 테이블 단일 진실화

이벤트 테이블을 통해서 세션의 모든 내용을 replay 방식으로 복구 가능하다는 것이 이 과제가 의도하는 event projection 방식의 핵심이라고 이해했습니다.

따라서 이벤트 테이블을 single source of truth 로 간주하고 append only, immutable 하도록 구현합니다.

### 메세지 수정 및 삭제

#### 구현한 방법

수정, 삭제된 메세지의 원본과 결과를 메세지 테이블에 저장할 것인지, 이벤트 테이블을 통해 조합해서 뷰를 생성하도록 할 것인지에 대해 다음과 같은 기준을 세웠습니다.

- 현재 상태의 메세지 조회는 메세지 테이블만으로 가능하도록 합니다.
- 특정 시점의 view 를 그려낼 때 이벤트 테이블을 사용합니다.

이러한 기준에 따라, 수정 및 삭제된 메세지를 다음과 같이 구현하여 메세지 테이블만으로 현재 상태를 조회 가능하도록 합니다.

- 수정
  - 메세지 수정본을 메세지 테이블에 업데이트하며, editedAt 컬럼에 수정일시를 저장합니다.
  - editedAt 값이 존재하는 메세지 옆에는 '수정됨' 으로 표시 가능합니다.
- 삭제
  - deletedAt 컬럼에 삭제일시를 저장하여 soft delete 로 구현합니다.
  - '삭제된 메세지입니다.' 처럼 placeholder 표시가 가능합니다.

#### 검토한 대안

- 수정
  - 메세지 테이블을 업데이트하지 않고 수정 이벤트를 저장하여, 이벤트를 이용해서 뷰를 그려낼 수 있습니다.
  - 현재 상태의 메세지를 조회할 때에도 이벤트 테이블을 조회해서 replay 해야 하므로, 대화가 쌓일수록 비용이 증가한다고 판단하였습니다.
- 삭제
  - 삭제된 메세지를 hard delete 하여 삭제된 레코드 및 deleted_at 컬럼 용량 절약이 가능합니다.
  - 이 경우, 삭제된 메세지의 placeholder가 보여지지 않게 됩니다.
  - 장단점의 크기가 기술적인 부분에서는 미미한 편이므로 ux 위주의 결정을 하였습니다.

## 기타 기준정의

### 이벤트, 메세지의 정렬 기준

이벤트와 메세지의 정렬 기준은 db가 생성하는 created_at 컬럼으로 지정했습니다.
낮은 확률로 같은 created_at이 발생한다면, id 컬럼을 보조적으로 사용합니다.

이로써 정렬 기준을 만드는 단일 권한은 db가 가집니다.

### 중복 이벤트 방지

클라이언트가 생성한 client_event_id 값을 중복 방지 키로 사용합니다.
동일한 메세지가 2회 수신되는 것은 다음 두 가지 상황이 있습니다.
- 사용자가 동일한 메세지를 2회 발신
- 원활하지 않은 웹소켓 메세지 전송으로 인한 중복 발신

서버는 이 2가지를 구분할 수 없습니다. 따라서 이를 구분하는 책임을 클라이언트에게 부여하는 것이 올바르다고 판단했습니다.

### 동일한 사용자쌍의 1세션 정책

같은 두 사용자가 동시에 진행 가능한 활성세션을 1개로 제한했습니다.
카카오톡 등 일반적인 1:1 채팅 UX와 일치합니다.

## 웹소켓 미구현

프로젝트 설정 시점에는 stomp 를 이용한 웹소켓 구현을 계획하고 의존성을 추가하였습니다.
disconnect, reconnect 기능을 구현하기에 앞서서 개발 완료 후의 테스트 방안을 생각하던 중, 웹소켓이 단순 pub/sub 메세지 중개 이상의 역할을 하지 않는다고 판단하여 제외하고 rest api로 갈음하였습니다.
상세 이유는 다음과 같습니다.

- 평가 영역
  - 본 과제의 평가 대상은 이벤트 소싱, 상태복원, 중복방지, 정합성이며, 완전한 제품 수준의 ux는 필수 구현사항이 아닙니다.
- 로직의 동일성
  - curl을 이용한 테스트 과정에서 rest api를 이용한 disconnect, reconnect 시뮬레이션 대비 웹소켓 방식이 갖는 특유의 목적이 없습니다.
- rest 만으로 검증 가능
  - 웹소켓을 구현하더라도 curl을 이용한 테스트를 위해 disconnect, reconnect 이벤트를 저장하는 rest api가 필요하며, 이를 이용해서 disconnect, reconnect를 시뮬레이션 가능합니다.

이 앱에서 사용자의 connection 상태는 `MockSessionManager.java` 클래스로 시뮬레이션하며, presence 판단에 사용합니다.
