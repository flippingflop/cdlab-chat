본 문서는 과제 4.2 의 "설계 문서에 포함되어야 할 항목" 5종을 정리합니다.

## 1. 재연결 시 데이터 정합성을 유지하는 방식

클라이언트의 연결이 끊어졌다가 다시 연결될 때, 끊긴 동안 발생한 변경이 누락 없이 클라이언트에 반영되는 메커니즘을 정리합니다.

웹소켓 미구현 결정에 대해서는 design-decision.md 에 정리되어 있습니다.
websocket 을 이용했다면 동일한 모델이 push 기반으로 동작했을 것이며, 현재 이를 REST 로 갈음한 상태입니다.

### 연결 끊김의 원인

연결 끊김의 원인은 다양합니다.

- 사용자의 네트워크 단절
- 클라이언트 앱의 백그라운드 전환
- 명시적 disconnect 호출 등

연결이 끊어진 동안 상대방은 메시지를 보내거나, 메세지를 수정/삭제하거나, 세션을 종료할 수 있습니다. 재연결 시 클라이언트는 끊긴 시점 이후의 모든 사실을 정확히 복원할 수 있어야 합니다.

### 이벤트 테이블의 활용

events 테이블이 single source of truth 입니다. append only, immutable 하므로 한번 저장된 레코드는 시간이 지나도 변하지 않습니다.

`SessionController.java`의 timeline api 는 특정 시점까지의 이벤트를 fold 하는 방식으로 replay하여, 해당 시점의 뷰를 반환합니다.

클라이언트는 재연결 후 timeline api 를 호출하여 화면을 그립니다. 끊긴 시점 이후의 모든 변화가 응답에 포함되며, 같은 시점에 대한 호출은 항상 동일한 응답을 반환합니다 (결정성).

### 재연결시 상태를 복구하는 방법

1. 클라이언트는 마지막으로 받은 이벤트의 시각 t 를 보관합니다.
2. 연결이 복구되면 `POST /api/sessions/{id}/connect` 를 호출합니다.
   - 서버는 events 에 (sessionId, userId, JOIN)가 일치하는 row 가 있으면 reconnect 를, 없으면 join 을 자동 발행합니다.
   - 클라이언트는 join/reconnect 를 명시적으로 발행하지 않습니다.
3. 클라이언트는 `GET /api/sessions/{id}/timeline?at=NOW` 를 호출하여 현재 상태로 업데이트 합니다.
4. 응답을 이용하여 화면을 갱신합니다.

이 시퀀스는 매 재연결마다 동일하게 동작합니다. 끊긴 시간이 길든 짧든 timeline 응답이 모든 사실을 포함합니다.

### WebSocket 운영 환경의 경우

websocket 을 채택했다면 다음과 같이 동작했을 것입니다.

- 클라이언트가 websocket 연결을 열 때 서버는 reconnect 이벤트를 발행하고, 마지막 조회 시각 이후의 이벤트를 push 합니다.
- 본 프로젝트의 timeline fold 로직을 그대로 재사용 가능합니다.

이벤트가 권위이므로 transport 가 push 든 pull 이든 결과는 동일합니다. 이 동질성이 WebSocket 미채택의 근거이기도 합니다.

### 정합성을 보장하는 방식

- 이벤트의 immutable + append-only: 끊긴 동안의 변화도 모두 이벤트에 영속됩니다. 사라지는 사실이 없습니다.
- timeline 결정성: 같은 (sessionId, t) 호출은 호출 시각과 무관하게 동일 응답입니다.
- client_event_id 멱등: 클라이언트가 메세지 발신 후 응답을 받지 못해 재전송하더라도 이벤트에 중복 row 가 쌓이지 않습니다.

### 한계 / 가정

- 서버 재시작 시 in-memory presence `MockSessionManager.java` 는 reset 됩니다.
- 실제 websocket 환경에서도 동일한 동작입니다. 이벤트는 DB 에 영속이므로 타임라인 복원에 영향을 주지 않습니다.
- 웹소켓 연결 상태를 조회하는 presence 자체는 휘발성 정보입니다. 특정 시점의 presence 는 이벤트에 남지만, 본 프로젝트의 타임라인 응답은 presence 를 포함하지 않습니다.
- presence 는 별도 endpoint `GET /api/sessions/{id}/presence` 로 현재 시점 스냅샷만 노출합니다. (웹소켓 연결 상태)

## 2. 중복 이벤트 / 메세지 처리 방식

같은 의미의 이벤트가 events 에 중복 저장되지 않도록 하는 방식을 정리합니다.

중복 발생 원인은 두 가지로 나누어 다룹니다.

- 클라이언트가 발행하는 이벤트의 중복 (네트워크 재시도, 사용자 중복 입력 등)
- 서버가 자체 발행하는 이벤트의 중복 (라이프사이클 신호의 재발행 등)

### 클라이언트 발행 이벤트의 중복 방지

클라이언트가 발행하는 이벤트인 message(작성, 수정 등), leave 는 client_event_id 라는 멱등 키를 가집니다.
같은 client_event_id 로 동일 요청이 두 번 들어오면, 두 번째 요청은 첫 번째와 동일한 응답을 반환하며 이벤트 테이블에 새 row 가 추가되지 않습니다.

검증 흐름은 다음과 같습니다.

1. 요청 수신 시 client_event_id 로 events 에 기존 row 가 있는지 선조회합니다.
2. 있으면 그 row 를 그대로 응답으로 변환하여 반환합니다.
3. 없으면 핸들러를 호출하여 이벤트를 발행합니다.

DB 의 client_event_id 컬럼에는 partial unique index 가 걸려 있습니다.
위 1번 선조회를 동시에 통과한 두 요청이 INSERT 단계에서 race 가 일어나면 한쪽만 성공하며, 실패한 쪽은 DataIntegrityViolationException 을 잡아 재조회로 동일 응답을 반환합니다.

선조회 및 db unique 제약조건을 이용해서 중복을 방지합니다.

### 서버 발행 이벤트의 중복 방지

서버가 자체 발행하는 이벤트인 SESSION_ENDED, JOIN, RECONNECT, DISCONNECT 는 client_event_id 가 NULL 입니다.
멱등 키가 없으므로, 서버가 "발행 시점의 상태" 를 기준으로 중복을 판단합니다.

- SESSION_ENDED — leave 이벤트로 인해 세션이 처음 ENDED 상태가 되는 시점에만 발행합니다. 이미 ENDED 상태에서 두 번째 leave 가 들어와도 SESSION_ENDED 는 다시 발생하지 않습니다.
- JOIN / RECONNECT — 이벤트 테이블에 (sessionId, userId, JOIN) 이 있으면 reconnect 를, 없으면 join 을 발행합니다. 클라이언트가 명시적으로 발행하지 않으므로 동일 사용자의 동시 connect 가 들어오는 race 자체가 발생하지 않습니다.
- DISCONNECT — `MockSessionManager.java` 에서 online 인 사용자만 발행 대상입니다. 이미 offline 인 사용자가 disconnect 를 호출하면 이벤트 테이블에 row 가 쌓이지 않습니다.

### 멱등의 단위

멱등은 "같은 client_event_id 가 같은 결과를 보장한다" 는 약속입니다.
클라이언트가 다른 client_event_id 로 같은 의미의 요청을 두 번 보내면, 두 요청 모두 별개의 이벤트로 events 에 저장됩니다. 멱등 단위 밖이기 때문입니다.

이는 의도된 정책입니다.
"같은 메세지를 두 번 보내고 싶었다" 와 "네트워크 재시도로 같은 요청이 두 번 도착했다" 를 서버가 구분할 수 없으므로, 그 구분을 client_event_id 발급 주체인 클라이언트에게 위임합니다. 같은 책임은 design-decision.md 의 "중복 이벤트 방지" 단락에서도 다룹니다.

## 3. 서버 수평 확장 시 세션 분산 / 상태 저장 전략

인스턴스를 여러 대로 확장할 때 고려할 사항을 정리합니다.

### 현재 가정

스프링부트 인스턴스 1대를 가정한 초기 비즈니스 앱입니다. design-decision.md 파일에도 이러한 가정이 명시되어 있습니다.

### 수평 확장이 수월한 영역

이벤트 테이블과 projection (messages, sessions) 은 모두 db 에만 상태를 가집니다. 인스턴스는 stateless 이므로 어느 인스턴스에 요청이 도달해도 동일한 결과를 반환합니다.
이러한 db 정합성 측면에서, 스프링부트 인스턴스를 추가하는 데 별도 작업이 필요하지 않습니다.

### 인스턴스 간 공유가 필요한 영역 — presence

현재 이 앱은 웹소켓의 작동을 `MockSessionManager.java`를 이용해서 in-memory 방식으로 시뮬레이션하도록 구현되어 있습니다.

만약 실제 웹소켓 환경인 경우, 외부의 redis 인스턴스를 분리하여 여러 스프링부트 인스턴스가 세션을 공유하도록 구현합니다.
presence 를 독립된 redis가 관리하므로, scale out 이 가능합니다. 

### websocket 도입 시 추가 고려

websocket 은 connection 자체가 인스턴스에 묶입니다.
사용자 A 가 인스턴스 1 에 connect 한 상태에서 사용자 B 가 인스턴스 2 에 메세지를 발신하면, 인스턴스 2 가 인스턴스 1 의 사용자 A 에게 push 해야 합니다.

이 경우 redis pub/sub 또는 메세지 브로커가 인스턴스 간 push 채널 역할을 합니다.

본 프로젝트는 REST 로 갈음했으므로 위 push 채널이 필요하지 않습니다. 클라이언트가 timeline api 를 polling 하거나 재연결 시 호출하는 방식으로 대체됩니다.

## 4. 관측 가능성 (로그 / 메트릭 / 추적)

운영 환경에서 시스템 상태를 외부에서 확인 가능하도록 하는 수단을 정리합니다.
현재는 hibernate 의 sql 로깅 외에는 별도의 관측 수단이 추가되어 있지 않습니다. 초기 비즈니스 앱 가정 단계이므로 우선 도입 영역에서 제외하였습니다.

### 도입 시 우선 고려할 지표

운영 도입 시 다음 지표들이 1차 관심사가 됩니다.

- events insert 발생률: 도메인 활동량의 1차 지표.
- timeline fold 비용: 세션 하나에서 진행되는 대화의 길이가 길어질수록 증가합니다. 스냅샷 도입 시점을 결정하기 위해 모니터링해야 합니다.
- 멱등성 문제 발생: client_event_id 선조회로 거른 비율입니다. 비정상적으로 높으면 클라이언트 재시도 로직을 점검할 필요가 있습니다.
- connect, disconnect, reconnect 빈도: 세션별/사용자별로 봅니다. 네트워크 품질 또는 클라이언트 안정성에 관한 지표입니다.

이외에도 웹 ui 를 필요로 한다면 datadog apm 을 도입할 수 있습니다.

## 5. 장애 대응 시나리오

운영 환경에서 발생 가능한 장애 유형별 대응 시나리오입니다.

### 서버 인스턴스 다운

이벤트 레코드가 db 에 영속되므로 도메인 사실은 영향받지 않습니다.
in-memory presence `MockSessionManager.java` 는 reset 됩니다. 실제 websocket 환경에서도 동일한 동작입니다.
사용자가 재연결하면 timeline 조회로 끊긴 시점 이후의 변화를 복원할 수 있으므로, 사용자 측 데이터 손실은 발생하지 않습니다.

### DB 장애

write 실패는 트랜잭션 단위로 throw 됩니다. 로직이 부분적으로 실패하는 경우(메세지는 저장됐는데 이벤트는 미저장 등)는 트랜잭션 롤백으로 일관성이 유지됩니다.
클라이언트는 응답을 받지 못하면 동일 client_event_id 로 retry 할 수 있습니다. retry 가 멱등이므로 db 가 복구되면 사용자 기능은 정상적인 흐름으로 돌아옵니다.

### 메세지 정합성 이상

메세지 테이블과 이벤트 테이블 사이에 불일치가 발견되더라도, events 가 single source of truth 이므로 messages 를 처음부터 다시 만들 수 있습니다.
이벤트를 시점순으로 fold 하면 동일한 메세지 결과가 재생성되므로, projection 손상은 회복 가능합니다.
