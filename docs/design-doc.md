## 재연결 시 데이터 정합성을 유지하는 방식

클라이언트의 연결이 끊어졌다가 다시 연결될 때, 끊긴 동안 발생한 변경이 누락 없이 클라이언트에 반영되는 메커니즘을 정리합니다.

웹소켓 미구현 결정에 대해서는 design-decision.md 에 정리되어 있습니다.
websocket 을 이용했다면 동일한 모델이 push 기반으로 동작했을 것이며, 현재 이를 REST 로 갈음한 상태입니다.

### 연결 끊김의 원인

연결 끊김의 원인은 다양합니다.

- 사용자의 네트워크 단절
- 클라이언트 앱의 백그라운드 전환
- 명시적 disconnect 호출 등

연결이 끊어진 동안 상대방은 메시지를 보내거나, 메세지를 수정/삭제하거나, 세션을 종료할 수 있습니다. 재연결 시 클라이언트는 끊긴 시점 이후의 모든 사실을 정확히 복원할 수 있어야 합니다.

### 이벤트 테이블의 활용

events 테이블이 single source of truth 입니다. append only, immutable 하므로 한번 저장된 레코드는 시간이 지나도 변하지 않습니다.

`SessionController.java`의 timeline api 는 특정 시점까지의 이벤트를 fold 하는 방식으로 replay하여, 해당 시점의 뷰를 반환합니다.

클라이언트는 재연결 후 timeline api 를 호출하여 화면을 그립니다. 끊긴 시점 이후의 모든 변화가 응답에 포함되며, 같은 시점에 대한 호출은 항상 동일한 응답을 반환합니다 (결정성).

### 재연결시 상태를 복구하는 방법

1. 클라이언트는 마지막으로 받은 이벤트의 시각 t 를 보관합니다.
2. 연결이 복구되면 `POST /api/sessions/{id}/connect` 를 호출합니다.
   - 서버는 events 에 (sessionId, userId, JOIN)가 일치하는 row 가 있으면 reconnect 를, 없으면 join 을 자동 발행합니다.
   - 클라이언트는 join/reconnect 를 명시적으로 발행하지 않습니다.
3. 클라이언트는 `GET /api/sessions/{id}/timeline?at=NOW` 를 호출하여 현재 상태로 업데이트 합니다.
4. 응답을 이용하여 화면을 갱신합니다.

이 시퀀스는 매 재연결마다 동일하게 동작합니다. 끊긴 시간이 길든 짧든 timeline 응답이 모든 사실을 포함합니다.

### WebSocket 운영 환경의 경우

websocket 을 채택했다면 다음과 같이 동작했을 것입니다.

- 클라이언트가 websocket 연결을 열 때 서버는 reconnect 이벤트를 발행하고, 마지막 조회 시각 이후의 이벤트를 push 합니다.
- 본 프로젝트의 timeline fold 로직을 그대로 재사용 가능합니다.

이벤트가 권위이므로 transport 가 push 든 pull 이든 결과는 동일합니다. 이 동질성이 WebSocket 미채택의 근거이기도 합니다.

### 정합성을 보장하는 방식

- 이벤트의 immutable + append-only: 끊긴 동안의 변화도 모두 이벤트에 영속됩니다. 사라지는 사실이 없습니다.
- timeline 결정성: 같은 (sessionId, t) 호출은 호출 시각과 무관하게 동일 응답입니다.
- client_event_id 멱등: 클라이언트가 메세지 발신 후 응답을 받지 못해 재전송하더라도 이벤트에 중복 row 가 쌓이지 않습니다.

### 한계 / 가정

- 서버 재시작 시 in-memory presence `MockSessionManager.java` 는 reset 됩니다.
- 실제 websocket 환경에서도 동일한 동작입니다. 이벤트는 DB 에 영속이므로 타임라인 복원에 영향을 주지 않습니다.
- 웹소켓 연결 상태를 조회하는 presence 자체는 휘발성 정보입니다. 특정 시점의 presence 는 이벤트에 남지만, 본 프로젝트의 타임라인 응답은 presence 를 포함하지 않습니다.
- presence 는 별도 endpoint `GET /api/sessions/{id}/presence` 로 현재 시점 스냅샷만 노출합니다. (웹소켓 연결 상태)
